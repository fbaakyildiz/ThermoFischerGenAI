Clinical AI research prototype

PCT-guided antibiotic stewardship, structured as a multi-agent review.

BrahmsAI turns structured patient data into a clinician-readable stewardship report. It combines B.R.A.H.M.S PCT threshold logic, serial PCT kinetics, safety overrides, prompt-injection defenses, output validation, and structured logging inside a local FastAPI application.

Project scope

A clinical decision-support concept with explicit guardrails.

The interface is built for local experimentation: enter patient context, choose a model provider, run the agent pipeline, and inspect the recommendation path. It is not a medical device and does not replace clinician judgment.

Method

Algorithm-aligned reasoning

PCT values are interpreted through pathway-specific LRTI and sepsis thresholds, including gray-zone handling and escalation logic for treatment monitoring.

Interface

Clinician-readable output

The final report surfaces recommendation strength, rationale, warnings, override flags, kinetic interpretation, token estimates, and latency metadata.

Security

Prompt-injection resistance

Clinical notes are sanitized before the model call, system prompts treat patient text as data only, and model responses are retried and validated as JSON.

Algorithm

PCT bands map the biomarker value to a constrained recommendation.

The model is not asked to invent clinical thresholds. It receives structured patient values and is instructed to apply the project threshold table for LRTI or sepsis.

  • LRTI: <0.10 withhold, 0.10-0.50 clinician decision bands, >0.50 start
  • Sepsis: <0.5 low risk, 0.5-2.0 indeterminate, >2.0 high risk
  • Gray-zone outputs require explicit clinician review

Multi-agent structure

Four agents, one final stewardship report.

Each stage has a narrow responsibility. That makes the reasoning easier to inspect and keeps the final report grounded in structured intermediate outputs.

A1

Intake and validation

Checks structured fields, normalizes context, and rejects clinical notes that look like instructions.

A2

Threshold reasoning

Applies PCT bands for the selected clinical setting and preserves gray-zone uncertainty.

A3

Kinetic and context review

Reviews serial PCT decline, antibiotic day, instability flags, and high-risk patient status.

A4

Report generation

Returns a structured report with recommendation, rationale, warnings, metadata, and review flags.

Engineering controls

Built for local use with explicit safety boundaries.

BrahmsAI focuses on the software architecture around clinical reasoning: schema validation, model routing, retry behavior, timeout control, structured logs, and cost estimates.

Model routing

The local UI can accept a model API key and route the pipeline to the compatible provider configured for the session.

  • Gemini, OpenAI, Claude, and OpenRouter style keys
  • Provider selection stays local to the running browser session
  • Timeout and JSON retry behavior around every agent call

Structured observability

Every pipeline run estimates token counts, latency, and cost so the local experiment remains auditable.

  • Per-agent metadata returned with the report
  • Pipeline-level totals for tokens, latency, and cost
  • JSON summary logs after completion

Run locally

Start the FastAPI app and open the local browser UI.

The project is intentionally local-first. API keys should be entered only in your own local environment and should not be committed to the repository.
